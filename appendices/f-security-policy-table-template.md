# 附录 F：安全策略表模板

本附录提供企业防火墙、交换机 ACL、安全组和零信任访问规则常用的安全策略表模板。安全策略表的作用是把“谁访问谁、用什么协议、为什么需要、是否记录日志、谁负责审批”写清楚。

安全策略不是简单的端口清单。一个合格的策略条目至少要包含源、目的、服务、动作、业务依据、日志要求和有效期。没有业务依据的放行规则，后期很难维护，也很难通过安全审计。

## F.1 策略编写原则

| 原则 | 说明 |
| --- | --- |
| 最小权限 | 只开放业务必需的源、目的和端口 |
| 默认拒绝 | 没有明确允许的流量默认拒绝 |
| 地址对象化 | 使用清晰的地址对象，避免直接散写大量 IP |
| 服务对象化 | 使用标准服务对象，避免 TCP/UDP 混乱 |
| 方向清晰 | 明确源区域和目的区域，不只写 IP |
| 记录日志 | 关键允许和拒绝策略应记录日志 |
| 定期复核 | 临时策略必须有到期时间 |
| 变更可追溯 | 每条策略关联工单、负责人和审批人 |

| 30 | allow-guest-to-internet-web | Guest | `OBJ_NET_GUEST` | Untrust | Any | DNS/HTTP/HTTPS | Allow | Source NAT | Enable | 访客上网 | 网络组 | CHG-2026-003 | 长期 |
| 40 | deny-guest-to-internal | Guest | `OBJ_NET_GUEST` | Internal | `OBJ_RFC1918_ALL` | Any | Deny | No | Enable | 阻断访客访问内网 | 安全组 | CHG-2026-004 | 长期 |
| 50 | allow-internet-to-dmz-web | Untrust | Any 或指定合作方 | DMZ | `VIP_PUBLIC_WEB_HTTPS` | HTTPS | Allow | DNAT/VIP | Enable | 外部访问官网 | 应用组 | CHG-2026-005 | 长期 |

填写建议：

- 策略名称使用动作、源、目的和服务组合，便于日志中识别。
- 源地址不要轻易写 Any。公网发布可根据业务情况限制合作方公网地址。
- 服务不要写 Any，除非是经过评审的特殊场景。
- 长期策略也要定期复核，不代表永久不检查。

## F.4 服务对象表

| 服务对象 | 协议 | 端口 | 用途 | 备注 |
| --- | --- | ---: | --- | --- |
| `SVC_DNS` | UDP/TCP | 53 | 域名解析 | 大响应可能使用 TCP |
| `SVC_HTTP` | TCP | 80 | Web 明文 | 通常跳转 HTTPS |
| `SVC_HTTPS` | TCP | 443 | 加密 Web | 证书和 SNI 另查 |
| `SVC_SSH` | TCP | 22 | 设备/服务器管理 | 只允许管理区 |
| `SVC_RDP` | TCP/UDP | 3389 | Windows 远程桌面 | 建议只经堡垒机 |
| `SVC_SNMP` | UDP | 161 | 监控轮询 | 优先 SNMPv3 |
| `SVC_SYSLOG` | UDP/TCP | 514 | 日志上送 | TCP 更可靠 |

服务对象要区分 TCP 和 UDP。把 DNS 只写成 TCP 53 或只写成 UDP 53，都可能导致部分场景异常。

| NAT ID | NAT 名称 | 类型 | 原始源 | 原始目的 | 原始服务 | 转换后源 | 转换后目的 | 出接口/区域 | 业务说明 |

NAT 策略和安全策略要成对检查。源 NAT 解决内网访问互联网的源地址转换，目的 NAT 解决公网访问内部服务器的地址发布，但它们通常不能替代安全策略。

## F.8 策略验证记录

| 验证项 | 验证命令或方法 | 预期结果 | 实际结果 | 结论 |
| --- | --- | --- | --- | --- |
| 源端测试 DNS | `nslookup intranet.example.com` | 解析到 `10.28.60.20` |  |  |
| 源端测试端口 | `Test-NetConnection 10.28.60.20 -Port 443` | TCP 成功 |  |  |
| 防火墙策略命中 | 查看策略 ID 命中数或日志 | 命中新增策略 |  |  |
| 防火墙 NAT 命中 | 查看 NAT 会话 | 转换符合预期 |  |  |
| 服务器日志 | Web/系统日志 | 看到源访问 |  |  |
| 反向验证 | 非授权源访问 | 被拒绝并记录日志 |  |  |

反向验证很重要。例如只允许办公网访问服务器时，应从访客网或未授权网段验证不能访问。否则容易只证明“允许有效”，没有证明“边界有效”。

| ---: | --- | --- | ---: | --- | --- |

| 20 | allow-temp-vendor-ssh |  |  |  | 到期删除 |

复核建议：

- 长期无命中的策略应确认是否可以删除。
- Any 源、Any 目的、Any 服务的策略必须重点复核。
- 临时策略到期后应删除或禁用。
- 离职人员、下线系统和过期公网地址要及时清理。

## F.10 常见错误

| 错误写法 | 风险 | 推荐做法 |
| --- | --- | --- |
| Office -> Server Any Allow | 办公网可访问所有服务器端口 | 按系统和端口拆分 |
| Any -> DMZ HTTPS Allow | 公网所有来源都能访问 | 根据业务限制来源，至少开启日志 |
| Guest -> Internal Allow | 访客可访问内网 | Guest 只允许 Internet，拒绝 RFC1918 |
| 管理口向全网开放 SSH | 设备暴露面过大 | 只允许管理区或堡垒机 |
| 临时策略无到期时间 | 长期遗留风险 | 写明到期日期和复核人 |

## F.11 自查题

1. 一条安全策略为什么不能只写“开放 443”？
2. 源 NAT、目的 NAT 和安全策略分别解决什么问题？
3. 为什么访客网要显式拒绝访问 RFC1918 私有地址？
4. 策略上线后为什么要做反向验证？
5. 哪些策略应该优先纳入定期复核？
