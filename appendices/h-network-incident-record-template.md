# 附录 H：网络故障处理记录模板

本附录提供网络故障处理记录模板。故障记录不是为了事后追责，而是为了把排错过程、证据、判断和恢复动作沉淀下来，避免同类问题反复发生。

一份好的故障记录应能回答：

- 故障什么时候开始，影响了谁。
- 现象是什么，哪些业务正常，哪些业务异常。
- 排查过哪些层次，证据是什么。
- 根因在哪里，如何恢复。
- 后续如何避免再次发生。

## H.2 故障级别定义

| 级别 | 定义 | 示例 | 响应要求 |
| --- | --- | --- | --- |
| P1 | 大范围核心业务中断 | 总部全员无法上网，核心系统不可用 | 立即响应，管理层同步 |
| P2 | 重要业务或区域中断 | 单个分支 VPN 中断，DMZ 重要系统不可访问 | 快速响应，业务负责人同步 |
| P3 | 局部影响，有替代方案 | 单个楼层无线异常，个别服务器端口不通 | 按运维流程处理 |
| P4 | 咨询、轻微异常或低风险事件 | 单个用户配置问题，日志告警需确认 | 常规处理 |

故障级别要结合业务影响判断，不只看技术现象。例如同样是 TCP 443 不通，如果影响的是核心交易系统公网入口，就可能是 P1 或 P2。

## H.3 影响范围记录

| 维度 | 内容 |
| --- | --- |
| 受影响用户 | 例如：总部办公网 VLAN 10 用户 |
| 受影响业务 | 例如：内部门户、OA、VPN、互联网访问 |
| 受影响地址段 | `10.28.10.0/24` |
| 受影响设备 | `CORE-SW-01`、`FW-HQ-01` |
| 不受影响范围 | 例如：服务器区内部互访正常，访客网上网正常 |
| 临时绕行方式 | 例如：切换备用出口、使用备用 VPN |

记录“不受影响范围”同样重要。它能帮助收敛故障边界，避免所有团队同时盲目排查。

| 09:20 | 确认客户端网关可达，DNS 正常 | 网络组 | 测试截图/命令输出 |
| 09:28 | 防火墙日志显示策略拒绝 | 安全组 | policyid 120 |

时间线要客观记录，不要事后只写“已处理”。发生过哪些判断和操作，后续复盘都要依赖时间线。

## H.7 故障边界判断

| 判断问题 | 结论 | 证据 |
| --- | --- | --- |
| 报文是否从客户端发出 | 是/否 |  |
| 报文是否到达网关 | 是/否 |  |
| 报文是否到达防火墙 | 是/否 |  |
| 防火墙是否允许 | 是/否 |  |
| NAT 是否按预期转换 | 是/否/不涉及 |  |
| 报文是否到达服务器 | 是/否 |  |
| 服务器是否响应 | 是/否 |  |
| 回包是否返回客户端 | 是/否 |  |

示例结论：

```text
客户端报文已到达防火墙，防火墙命中允许策略并转发到 DMZ。
服务器未返回 SYN-ACK，故障边界收敛到服务器监听、本机防火墙或服务器回程路径。
```

## H.8 根因分析

### 根因分类

| 分类 | 示例 |
| --- | --- |
| 配置变更 | 策略顺序错误、路由缺失、VLAN 未放行 |
| 设备故障 | 接口异常、模块故障、电源故障 |
| 链路故障 | 运营商中断、光模块异常、线缆松动 |
| 容量问题 | 带宽打满、会话数耗尽、CPU 高 |
| 安全拦截 | IPS/URL 过滤/AV 误拦截 |
| 服务器问题 | 服务未监听、本机防火墙、网关错误 |
| 外部依赖 | DNS、证书、第三方接口、云服务异常 |
| 操作流程 | 未评审、未备份、未通知、回退不完整 |

### 根因描述

| 项目 | 内容 |
| --- | --- |
| 直接原因 |  |
| 深层原因 |  |
| 触发因素 | 例如：09:00 策略变更、链路切换、证书更新 |
| 为什么未提前发现 |  |
| 为什么监控或流程未阻止 |  |

根因描述要避免模糊词。例如“网络问题”不是根因；“防火墙策略 `deny-guest-to-internal` 顺序被错误上移，覆盖了办公网到服务器区 HTTPS 允许策略”才是可复盘的根因。

## H.11 沟通和通报模板

### 初始通报

```text
【故障通报】
时间：YYYY-MM-DD HH:MM
现象：总部办公网用户访问内部门户 HTTPS 失败。
影响范围：初步判断影响 VLAN 10 办公网，其他业务待确认。
当前进展：网络组、安全组已开始排查，下一次更新将在 30 分钟内同步。
```

### 恢复通报

```text
【故障恢复通报】
时间：YYYY-MM-DD HH:MM
现象：总部办公网访问内部门户已恢复。
影响时长：约 XX 分钟。
处理措施：已回退本日新增防火墙策略顺序调整。
后续安排：继续观察 24 小时，并完成根因复盘和整改。
```

### 复盘结论通报

```text
【故障复盘结论】
根因：防火墙策略顺序调整错误，导致办公网到服务器区 HTTPS 被拒绝。
改进措施：
1. 新增策略上线前执行策略匹配测试。
2. 关键业务策略变更必须双人复核。
3. 增加办公网到核心业务系统的主动探测监控。
```
